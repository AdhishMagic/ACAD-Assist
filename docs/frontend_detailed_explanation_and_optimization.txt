ACAD-Assist Frontend Detailed Explanation and Optimization Guide
Date: 2026-03-30
Scope: frontend source architecture, page-level responsibilities, duplicate reduction, and separation of concerns

============================================================
1) HOW THE FRONTEND CURRENTLY RUNS (END-TO-END FLOW)
============================================================

1. Entry point
- File: frontend/src/main.jsx
- What it does:
  - Bootstraps React with StrictMode.
  - Wraps app with Redux Provider and React Query QueryClientProvider.
  - Uses BrowserRouter for route handling.
- Why this matters:
  - This file is the root composition point for app-wide providers and app-wide behavior.

2. App shell handoff
- File: frontend/src/App.jsx
- What it does:
  - Renders AppRoutes only.
- Why this matters:
  - Keeps app shell minimal and delegates navigation responsibilities to routes.

3. Route orchestration
- File: frontend/src/routes/AppRoutes.jsx
- What it does:
  - Defines all route trees for guest/authenticated users and role-based routing.
  - Wraps protected areas with ProtectedRoute and RoleGuard.
  - Maps route constants for teacher/hod/admin sections.
- High-impact issue found:
  - Duplicate route declaration exists for /system/dashboard (declared twice).

4. Authentication and role guarding
- File: frontend/src/routes/ProtectedRoute.jsx
  - Enforces isAuthenticated before rendering protected outlet.
- File: frontend/src/routes/RoleGuard.jsx
  - Enforces allowed roles and redirects unauthorized users.
- File: frontend/src/routes/RoleBasedRedirect.jsx
  - Converts active role to dashboard path.

5. Layout layers
- Dashboard layout: frontend/src/layouts/DashboardLayout/DashboardLayout.jsx
  - Main authenticated shell (sidebar, navbar, right panel, outlet).
  - Contains UI state for sidebar collapse/mobile and dark mode.
- Auth layout: frontend/src/layouts/AuthLayout/AuthLayout.jsx
  - Split screen branding + auth form outlet.
- Main layout: frontend/src/layouts/MainLayout/MainLayout.jsx
  - Generic marketing shell, currently not the main route wrapper in AppRoutes.
- Legacy/extra layout: frontend/src/layouts/AppLayout/index.jsx
  - Appears to be another dashboard-like shell and risks duplication with DashboardLayout.

6. Global state and reducers
- File: frontend/src/store/store.js
  - Configures Redux store with rootReducer and apiMiddleware.
- File: frontend/src/store/rootReducer.js
  - Combines auth, courses, ai, knowledge, qpaper slices.
- Important inconsistency:
  - Two auth slices exist:
    - frontend/src/features/auth/store/authSlice.js (the one used by rootReducer)
    - frontend/src/features/auth/authSlice.js (parallel implementation)

7. API layer
- Central api client: frontend/src/services/api/axios.js
- Feature api clients:
  - frontend/src/services/api/auth.api.js
  - frontend/src/services/api/courses.api.js
  - frontend/src/services/api/ai.api.js
  - frontend/src/services/api/rag.api.js
  - frontend/src/services/api/qpaper.api.js
- Duplicate concern:
  - Additional per-feature API files under feature folders also create axios instances and mock logic, creating multiple transport styles.

3) DUPLICATE CODE / ARCHITECTURE GAPS FOUND
============================================================

1. Duplicate route entries
- /system/dashboard appears twice in AppRoutes.

2. Duplicate auth slice implementations
- features/auth/store/authSlice.js and features/auth/authSlice.js.
- Root reducer currently uses store/authSlice, so the other is likely dead code.

3. Duplicate page modules
- Legacy auth pages under src/pages/auth and active auth pages under src/features/auth/pages.
- Duplicate settings page under src/pages/settings and src/features/settings/pages.
- Multiple NotFound pages.

4. Multiple API client patterns
- services/api/* has one style.
- feature-specific service files create additional axios clients and independent mock interceptors.
- Different env vars used: VITE_API_BASE_URL vs VITE_API_URL.

5. Mock data spread across service files
- teacherAPI, hodAPI, adminAPI, notesAPI, settingsAPI, aiAPI all include mock logic inline.
- This mixes transport, fallback policy, and mock data in one place.

6. UI duplication in table/list pages
- Teacher/HOD/Admin pages repeat the same table shell + filter toolbar + empty state patterns.

7. Layout overlap
- DashboardLayout and AppLayout both represent dashboard shells.
- MainLayout exists but not strongly integrated in route tree.

============================================================
4) SEPARATION OF CONCERNS TARGET DESIGN
============================================================

A. Presentation layer
- Keep pages as composition-only shells.
- Move business/data logic into:
  - hooks (useXxxQuery/useXxxMutation/useXxxFilters)
  - services/api adapters
  - redux slices only when truly global

B. Domain feature boundaries
- Keep each feature self-contained:
  - features/<domain>/pages
  - features/<domain>/components
  - features/<domain>/hooks
  - features/<domain>/api
  - features/<domain>/mappers
  - features/<domain>/types

C. Shared layer boundaries
- shared/ui: reusable visual components only
- shared/lib/http: one axios base client + auth refresh interceptor
- shared/lib/query: shared React Query key factories and utilities
- shared/utils: pure helpers only (no side effects)

D. Mock strategy
- Create one mock adapter switch by environment:
  - VITE_API_MODE=real|mock
- Keep mock payloads in fixtures files, not in service functions.

5) OPTIMIZATION PLAN (NO DUPLICATES)
============================================================

Phase 1: Safe cleanup (no behavior change)
1. Remove duplicate /system/dashboard route declaration.
2. Remove or archive unused page stubs:
   - src/pages/auth/*
   - src/pages/settings/SettingsPage.jsx (if unused)
   - any unreferenced page shells.
3. Keep only one auth slice implementation (recommended: features/auth/store/authSlice.js).
4. Keep one NotFound page and route all 404s to it with optional role-aware copy.

Phase 2: API unification
1. Define a single http client in services/api/axios.js.
2. Convert feature service files to use that client only.
3. Standardize env var naming:
   - Use VITE_API_BASE_URL, VITE_AI_SERVICE_URL, VITE_RAG_SERVICE_URL consistently.
4. Move fallback/mock logic to a separate adapter layer.

Phase 3: Reusable UI patterns
1. Extract shared data table shell component with:
   - header actions slot
   - filter toolbar slot
   - loading state
   - empty state
2. Reuse across admin/teacher/hod list pages.
3. Extract shared analytics card and chart wrappers.

Phase 4: Hooks and query separation
1. Add hooks per domain:
   - useAuth, useCourses, useTeacherDashboard, useAdminUsers, useHodApprovals, etc.
2. Keep page files under 150 lines where possible.
3. Pages orchestrate only:
   - read URL params
   - call hooks
   - render components

============================================================
6) CODE-LEVEL GUIDELINES TO PREVENT FUTURE DUPLICATION
============================================================

1. One source of truth for each concern:
- One auth slice.
- One API client base.
- One route registry per role domain.

2. No axios.create inside page-specific service files unless domain-isolated and justified.

3. No inline mock datasets in production service files.
- Keep fixtures in a dedicated mock folder.

4. Route constants and links must use centralized route generators.
- Prevent hardcoded path drift between sidebar and AppRoutes.

5. Keep feature modules vertically sliced.
- Do not place active feature pages under src/pages and src/features simultaneously.

============================================================
7) HIGH-PRIORITY FILES TO REFACTOR FIRST
============================================================

1. frontend/src/routes/AppRoutes.jsx
- Remove duplicate route declaration and simplify route tree.

2. frontend/src/features/auth/store/authSlice.js + frontend/src/features/auth/authSlice.js

5. frontend/src/features/hod/services/hodAPI.js

============================================================
8) SUMMARY
============================================================

Your frontend already has good domain segmentation by role/features, but duplication exists in routing, auth state, API clients, and legacy page stubs.

To optimize with no duplicates and clean separation of concerns:
- Keep one routing map per role and one fallback page.
- Keep one auth slice and one API client strategy.
- Move mock data and fallback rules out of service methods.
- Convert repetitive page tables/forms/charts into reusable feature-agnostic components.
- Keep pages thin and push logic into hooks/services.

This document is now your baseline refactor map for frontend cleanup and scalability.
